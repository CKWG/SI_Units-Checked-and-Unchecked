Execution Time Measurements

Checking units of measurement during runtime will inevitably slow down the execution time compared to the pure numerics, a waste (?) of execution time which may be prohibitive for some applications. This library enables using strict unit checking during the development phase (until unit test phase), and switching off the checks when releasing the project into the wild. Nevertheless, there will be residues of the checking in the code which slow down the execution time, see the relative timing below in the comparison of the Nude and the Optimized values in the Unchecked variant, which represent the truly important results.

So before you now delve into the measurement results because you are pondering the idea of unit checking, keep in mind that the results presented here are not representative. Do measure your project yourself!

If you feel this library inappropriate, think about trying one of these alternatives:

  • Dmitry Kazakov's Units of Measurement for Ada
  • AdaCore's language extension (see chapter 6.5) of which I gave a (meanwhile outdated) assessment here: Physical units with GNAT (I do not know if there has been any progress, I have never used it myself – but note: This is not Ada standard and probably never will be).

The Schottky Langmuir equation is used for execution time measurements because it has rational exponents. Quaternions may be used similarly to matrices for coordinate transformations but need much more time.

Measurements of relative speeds were performed with:

  • GNAT CE 2021 on Windows 8.1 with an old and slow machine,
  • GNAT CE 2021 on Windows 11 with a brandnew fast machine;

Unfortunately, there is a bug in GNAT 15.2.1 (reported 05.11.2025, Bugzilla 122574): The switch -gnatn for inlining prevents linking because it produces unresolved references. Inlining function calls is essential for this library – without it, execution time literally explodes. The correction is expected in the next GNAT generation 16.

Loops are used with one million repetitions.

Assertion Effect

Before the measurements are discussed, a strange effect has to be discussed. We are interested in the speed of the checked and unchecked versions in comparison with the pure mathematics, i. e. the time consumed for the nude Schottky Langmuir equation with only floating point data. For the different Ada code versions of the equation used for measurements see below.

Thus, first the effect of switching on and off the Assertion_Policy pragma on the nude equations is measured:

This nude equation is compiled twice, once with Assertion Policy set to Ignore and once set to Check. Although the code is identical and there are no assertions, the execution times differ by an astonishing big factor. This difference must solely be attributed to the differences in the executables produced by GNAT depending on the Assertion_Policy pragma, which corresponds to the switch -gnata.

Comparing the above pair slow vs. fast machine, you see a speed-up with the new fast machne by a factor of about 5.5, whereas the relative time differences remain approximately the same between 6 and 7.

Schottky Langmuir

The Schottky Langmuir equation
        
is ideal for measurements: It has a lot of dimensioned items, a root, and rational exponents. The reference is the evaluation time of the nude equation, i.e. the pure floating point version.

In contrast, the dressed version is correctly dimensioned and called with parameters dimensioned with unit strings. Unit string evaluation is time consuming and heavily depends on the complexity of the string. However, unit strings are generally only used for initialization and do not normally occur in formulas. Thus in the stringless version, the parameter unit initialization is taken out of the loop.

Last, the optimized version also precomputes the constant part outside the loop. This is what an optimizing compiler will do (in the nude version), but cannot here because of the dimension components. (Do not confuse the optimized equation with the -O3 optimized executable.)

  • Dressed: complete evaluation inside loop
  • Stringless: unit strings precomputed outside loop
  • Optimized: constant part precomputed outside loop
  • Nude: pure numerics

These four versions (for details, see the program measure.adb in directory Examples) are computed once with unit checking enabled and once disabled. The results are quite astonishing. (Of course the absolute values depend on the speed of the machine in use; important are the relative speed values only.)

The relative factor between Nude and Optimized is approximately independent of the machine, 11 resp. 12 in the unchecked version versus 54 resp. 48 in the checked variant, so switching off the dimension check is quite effective: about 50 to 10 compared with the nude equation. A slow unit testing is tolerable. You have to decide by your own measurements whether the residue of 10 in the Unchecked case is tolerable for your application. These figures are in effect the most interesting ones.

Comparing the stringless and the dressed execution times, you can see that the unit string evaluation is extremely expensive in the Unchecked version, thus be careful to only use them for data initialization and avoid them in calculations. In the Checked version, this effect is smaller since the computation of the dimension components consumes most of the time, whereas in the unchecked version, only the dimension string evaluation is left.

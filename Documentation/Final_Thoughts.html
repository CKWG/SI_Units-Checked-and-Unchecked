Final Thoughts

This library has been, from the very start on in 2002, a veritable challenge for GNAT, the only compiler I ever have worked with privately – there are still some work-arounds for Gnat bugs left in the code. I by now deem the library nearly perfect; it has been thoroughly tested. So further releases will most probably only be bug fixes of my code or work-around removals.

I do not see important features missing. In the Vector_Space package, very few subtypes with predicate failures for vectors and matrices have been defined. If you miss some, you are kindly invited to send me a note with the missing definitions and perhaps a very simple example for the Examples directory. I will add them together with your name as author.

Celsius Degrees

Celsius temperatures have purposely been defined separately from SI items with different syntax. Of course transformations to and from Kelvin are provided. They also have to be read and written with a dedicated package of their own.

A possible library enhancement could be the possibility to directly read (nota bene: not to write) Celsius degrees with the IO package for items, but to automatically return them converted in Kelvin. Admittedly, this would not save much work because when reading from files, you normally know what to read and can easily use the appropriate package and convert the result.

Any thoughts?

Input from Files

When you develop with the checked variant and have 100% test coverage, you can be certain that there is no dimension error lurking in your code so that you can safely switch to the unchecked variant for deployment.

There is however one hole in this reasoning: input from external devices! When you read data with the wrong dimension, there is no way in the unchecked variant to prove that the input is correct. (This is like outputting a distance with a dimension format modifier for seconds – also this cannot be caught in the unchecked variant.)

This input deficiency can be remedied by also adding a dimension format modifier for the Get operations:

  procedure Get (File : in  File_Type;
                 X    : out Item;
                 Width: in  Field  := 0;
                 Dim  : in  String := "");

The dimension format modifier, if given, will ignore any prefixes and verify the unit string input, raising Unit_Error in case of failure. For example, a modifier "m/s" will check that the input has dimension Length/Time. This is not a restriction since normally you have to know what to read and can select the proper dimension. If for some reason, you do not know the expected dimension (when for instance reading table data), you simply omit the parameter. (The modifier will of course be ignored in the checked variant.)

Safety will cost execution time!

For the implementation, a clone of the item's dimension component will be needed.

Any thoughts?

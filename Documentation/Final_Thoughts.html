Final Thoughts

This library has been, from the very start on in 2002, a veritable challenge for GNAT, the only compiler I ever have worked with privately – there are still some work-arounds for Gnat bugs left in the code. I by now deem the library nearly perfect; it has been thoroughly tested. So further releases will most probably only be bug fixes of my code or work-around removals.

I do not see important features missing. In the Vector_Space package, very few subtypes with predicate failures for vectors and matrices have been defined. If you miss some, you are kindly invited to send me a note with the missing definitions and perhaps a very simple example for the Examples directory. I will add them together with your name as author.

Input from Files

I ponder the idea of one perhaps important addition:

When you develop with the checked variant and have 100% test coverage, you can be certain that there is no dimension error lurking in your code so that you can safely switch to the unchecked variant for deployment.

There is however one hole in this reasoning: input from external devices! When you read data with the wrong dimension, there is no way in the unchecked variant to prove that the input is correct. (This is like outputting a distance with a dimension format modifier for seconds – also this cannot be caught in the unchecked variant.)

This input deficiency can be remedied by also adding a dimension format modifier for the Get operations:

  procedure Get (File : in  File_Type;
                 X    : out Item;
                 Width: in  Field  := 0;
                 Dim  : in  String := "");

The dimension format modifier, if given, will ignore any prefixes and verify the unit string input, raising Unit_Error in case of failure. This is not a restriction since normally you have to know what to read and can select the proper dimension. If for some reason, you do not know the expected dimension (when for instance reading table data), you simply omit the parameter. (The modifier will of course be ignored in the checked variant.)

For example, a modifier "m/s" as well as "km/h" will check that the input has dimension Length/Time.

Safety will cost execution time!

For the implementation, a clone of the item's dimension component will be needed.

Any thoughts?

Tables

The Get procedure of RM A.10.9(12) offers the parameter Width that defines how many characters are read from the file:

  procedure Get (File : in  File_Type;
                 Item : out Num;
                 Width: in  Field := 0);

The default 0 is for reading syntactically correct input as long as possible. For a nonzero value, Get reads exactly so many characters (or up to a line terminator, whichever comes first) irrespective of the syntax and evaluates the input afterwards. Thus this option is ideal for reading tabular data. Width corresponds roughly to the sum of Fore, Aft and Exp of the corresponding Put procedure. Such a table might look like this (the first line is not data, it just shows the layout):

  12345678901234567890--123456789012345----123456789012345
  Earth Mass                 5.9722E+24    kg
  Equatorial Radius           6378.1370    km
  Polar Radius                6356.7523    km
  Mean Radius                 6371.0088    km
  Gravitational Const.      6.67430E-11    m**3*/(kg*s**2)
  Acceleration                   9.8203    m/s**2

The first block with the data description occupies the first 20 columns; a padding of 2 blanks follows as a block separator; then Width = 15 characters of numeric data, right-aligned in the second block; another padding of 4 blanks; the physical dimension is given in the third block (left-aligned as shown here or right-aligned). The first block und following padding are of no interest here; we're interested solely in the two data blocks and the separating padding.

As can easily be seen, such a table is impossible to write or read with the current SI IO features; especially the Width parameter is completely useless.

In directory Examples, procedure Tables writes and reads two tables, one with data left-aligned starting with column 24:

  123456789012345678901234
  Earth Mass             5.9722E+24*kg
                         5.9722E+0*Rg
  Equatorial Radius      6378.1370*km
  Polar Radius           6356.7523*km
  Mean Radius            6371.0088*km
  Gravitational Constant 6.67430E-11*m**3*kg**(-1)*s**(-2)
  Acceleration           9.8203*m/s**2

the other also starting at the same column (denoted by '|'), but shifted so that units are aligned at column 13 (counting from '|' as 1) in the second block denoted by '*':

  12345678901234567890123|23456789012*
  Earth Mass               5.9722E+24*kg
                               5.9722*Rg
  Equatorial Radius         6378.1370*km
  Polar Radius              6356.7523*km
  Mean Radius               6371.0088*km
  Gravitational Constant  6.67430E-11*m**3*kg**(-1)*s**(-2)
  Acceleration                 9.8203*m/s**2

Both are not what one would expect for a pleasant table. Also here, the Width parameter is useless!

A better implementation is needed for a nonzero value of Width.

So in order to split the numeric data and the unit into two blocks, the Put and Get operations will need additional parameters like the following. (Note that because of the defaults legacy code is not affected by this change.)

  procedure Put (File: in File_Type;
                 X   : in Item;
                 Fore: in Field  := Default_Fore;
                 Aft : in Field  := Default_Aft;
                 Exp : in Field  := Default_Exp;
                 Dim : in String := "";
                 Pad : in Field  := 0;
                 Unit: in Field'Base := 0) with
    Pre => (if Unit = 0 then Pad = 0);

If the new parameters Pad and Unit are zero, the resulting output will be the same as currently: the numeric value output according to Fore, Aft, Exp, the unit following as requested by Dim: -9.82*m/s**2

For a nonzero value of Unit, however, it is the duty of the programmer to choose the values for Fore, Aft and Exp such that the numbers are right-aligned. The numeric value will be written as before to the first table block, followed by Pad spaces for the padding, then the dimension string will be written to the second table block with abs Unit characters, right-aligned for a positive value, left-aligned for a negative one (without the leading '*', but with the leading '/') like in the very first example above.

The corresponding Get operation also needs the same new parameters Pad and Unit (ignore the Dim parameter for now – it's only there as a reminder that the idea of unit input checking of above might be implemented):

  procedure Get (File : in  File_Type;
                 X    : out Item;
                 Width: in  Field  := 0;
               [ Dim  : in  String := ""; ] 
                 Pad  : in  Field  := 0;
                 Unit : in  Field  := 0) with
    Pre => (if Width = 0 then Pad = 0 and Unit = 0);

If Width = 0, the result will be the same as currently, reading continues as long as the syntax allows.

For a nonzero value of Width, for all three parameters Width, Pad, Unit, exactly so many characters will be input or the characters (possibly none) up to a line terminator, whichever comes first, then reading stops. Then the three fields will each be evaluated. Contrary to the current beviour, Width applies only to the numeric value of X as prescribed in RM A.10.9(19). Data_Error will be raised if any of the Pad characters is a nonblank character. Illegal_Unit will be raised if the unit sequence does not have the syntax of a dimension format modifier after stripping any leading and trailing blanks (thus there is no difference in reading left-aligned or right-aligned unit strings).

Returns in X the (dimensioned) Item.

Proposals for other parameter profiles?

Any thoughts?

Celsius

Celsius degrees are an addition that can easily be omitted from any application, just remove its packages.

As currently defined, they have only limited operations, for instance they cannot be added, differences are measured in Kelvin. This might be too limiting. However, the to_Kelvin function has to be able to discriminate between temperatures and their differences. Thus, the model to define a Celsius algebra is Ada's handling of time and duration. Differences must be measured in a separate type, let us call it Diff; its conversion to Kelvin keeps the numeric value, its unit symbol may also be °C.

This leads to additional functions like

Celsius - Celsius → Diff
Celsius + Diff    → Celsius

I do not see this as a relevant feature. If you do need it, do not forget to define an appropriate IO functionality for Diff.

GNAT Bugs

This is a list of GNAT bugs affecting the code.

  • Known bug in GNAT CE 2021:
    The wrong exception Assertion_Error (expected: Unit_Error) is raised for an assignment with an incompatible dimension like 
        declare
      X: Time;  -- := 1.0*"m";  -- correct: Unit_Error
    begin
      X := 1.0*"m";  -- wrong: Assertion_Error
    end;
    For a wrong initialization, the correct exception is raised.
    What is astonishing is that exactly this code is used in function SI_is_Unchecked, where it works correctly.
    This bug does not occur in GNAT 15.2.1.

    reported 26.05.2020: fixed in GNAT 15.2.1; [T520-013 public] did not fix it.

    reported 04.10.2025: being fixed.
